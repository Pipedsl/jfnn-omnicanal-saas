## BUG-W06 — Re-engage archiva venta anterior con fecha de hoy, inflando KPIs de ventas diarias

**Severidad:** Crítica (corrompe métricas de ventas — venta de abril aparece como venta de hoy)
**Componente:** `sessions.service.js` → flujo re-engage → INSERT en `pedidos` + `getDashboardMetrics`

**Síntoma:**
Cuando Carlos (+56974792499) escribió hoy (2026-05-04), el sistema detectó que tenía una sesión en ENTREGADO del 14-04-2026 y la archivó. La query de métricas usa `archivado_en` para filtrar rango "hoy":
```sql
DATE(archivado_en AT TIME ZONE 'America/Santiago') = DATE(NOW() ...)
```
Resultado: venta de abril aparece como "1 venta hoy $25.000", con conversión 100%.

**Causa raíz:**
El re-engage hace `INSERT INTO pedidos (..., archivado_en) VALUES (..., NOW())`. La fecha de archivado queda en el día del re-engage, no en la fecha original de la venta.

**Fix:**
Al archivar por re-engage, usar la fecha original de la sesión como `archivado_en`:
```js
archivado_en: session.updated_at || session.created_at  // fecha real de la venta
```
O agregar una columna `re_engage_en` para distinguir el archivado por re-engage del archivado real, y excluirlo de las métricas de ventas diarias.

## BUG-W07 — Precios pierden separador de miles — cliente recibe monto de transferencia incorrecto

**Severidad:** Crítica (el cliente recibe el monto de transferencia equivocado — $167 en vez de $167.000)
**Componente:** QuoteCard input de precio → mensaje de cotización

**Síntoma (cotización mecánico JFNN-2026-1393A4):**
```
$18 c/u  (debería ser $18.000)
$85 c/u  (debería ser $85.000)
$32 c/u  (debería ser $32.000)
TOTAL: $167  (debería ser $167.000)
```
Y en el mensaje de transferencia: `"Transfiere los $167 a Scotiabank..."` — Pedro podría transferir exactamente $167 y el pago sería rechazado.

**Causa probable:**
El campo de precio usa `parseFloat()` o similar para leer el valor ingresado. En Chile el separador de miles es el punto (`18.000`), pero `parseFloat("18.000")` retorna `18.0` → se pierden los tres ceros.

**Fix:**
En el input de precio del QuoteCard, antes de enviar, limpiar el punto de miles:
```js
const precio = parseInt(value.replace(/\./g, ''), 10);
```
O usar un input numérico que no acepte puntos como separador decimal.

**Flujo deseado:**
1. Durante PERFILANDO, una vez que el agente tiene vehículo + repuesto(s), **antes de pasar a ESPERANDO_VENDEDOR**, preguntar:
   > "¿Cómo prefiere recibir su pedido: retiro en sucursal o envío a domicilio?"
2. Si elige **retiro**: preguntar "¿Melipilla o San Felipe?"
3. Capturar como entidad `sucursal_retiro` ('Melipilla' | 'San Felipe') y `metodo_entrega` ('retiro' | 'domicilio')
4. En la bandeja del dashboard, mostrar la sucursal de destino → el vendedor de cada sucursal filtra sus propias cotizaciones

**Regla provisional (primeros meses de operación San Felipe):**
- Clientes que piden **domicilio** → asignar a sucursal Melipilla (vendedores nuevos de San Felipe no manejan despacho aún)

- **Vendedor San Felipe** → ve solo cotizaciones de su sucursal

Esto implica que el usuario del dashboard debe tener un `rol` (`vendedor_melipilla`, `vendedor_san_felipe`, `admin`) y la bandeja filtra automáticamente según ese rol.

**Nota:** Este REQ es prioritario sobre REQ-02 completo — se puede implementar parcialmente capturando `sucursal_retiro` en entidades JSONB (sin migración) y mostrándolo en la bandeja como indicador visual, aunque el filtro automático por vendedor quede para REQ-02.

## BUG-POST03 — Repuesto "filtro de aceite" aparece sin que el cliente lo solicitara

**Severidad:** Alta (cotización incluye productos que el cliente no pidió → riesgo de cobro indebido o confusión)
**Detectado:** 2026-05-05
**Componente:** `backend/services/sessions.service.js` — flujo de re-engage / merge de entidades

**Síntoma:**
Cliente Carlos `+56974792499` (cliente recurrente con sesión ENTREGADA del 2026-05-04) escribió hoy:
> "Hola, necesito pastillas de freno para Toyota Corolla 2018"

Su mensaje SOLO menciona pastillas de freno. Sin embargo, el QuoteCard mostró 2 repuestos:
- `filtro de aceite` (sin precio)
- `pastillas de freno` (sin precio)

El "filtro de aceite" proviene de la sesión anterior de Carlos (E2E del 04 may donde cotizó filtro de aceite + filtro de aire para el mismo vehículo).

El re-engage al detectar sesión ENTREGADA antigua y crear una nueva sesión, está mergeando `repuestos_solicitados[]` viejos con los nuevos en lugar de empezar limpio. La nueva sesión debería arrancar solo con lo que el cliente menciona en su mensaje actual.

**Fix sugerido:**
- Al detectar re-engage (cliente con sesión ENTREGADA + nuevo mensaje), archivar la sesión vieja y crear una sesión NUEVA con `entidades.repuestos_solicitados = []` (vacía). El agente reconstruye desde cero según el nuevo mensaje del cliente.
- Mantener historial del cliente en memoria del agente para sugerencias ("¿Te traemos también filtro de aceite?") pero NO copiarlos automáticamente al QuoteCard.

## REQ-07 — Optimización de tokens de la IA (reducción de costo por conversación)

**Prioridad:** Alta (costo operativo creciente conforme sube el volumen)
**Detectado:** 2026-05-05
**Tipo:** Requerimiento de optimización / sostenibilidad económica

**Problema:**
El costo por conversación con la IA (Gemini Pro/Flash) está siendo demasiado alto en relación al ticket promedio del negocio. Cada turno de conversación hace una llamada completa al LLM enviando: system prompt extenso + base de conocimiento + historial + JSON schema. Para un cliente que requiere 8-12 turnos antes de cerrar venta, los costos se acumulan.

**Diagnóstico inicial (a confirmar con métricas):**
Los componentes que más tokens consumen por llamada:
1. **System prompt completo** en cada turno (~3-5k tokens) — incluye reglas duras, base de conocimiento, ejemplos, esquema JSON
2. **`knowledge-base.md` completo** (~2k tokens) — se inyecta en cada llamada aunque el 80% no aplique al estado actual
3. **`knowledge.json` (HU-7)** dinámico — crece con el tiempo
4. **Historial de mensajes** — enviado completo aunque solo los últimos 3-4 turnos sean relevantes
5. **JSON schema de output** detallado — repetido en cada llamada
6. **Routing Flash vs Pro** existe pero hay que verificar si se usa correctamente (Pro es ~10x más caro)

**Estrategias de optimización (ordenadas por impacto estimado / esfuerzo):**

### A. Context caching de Gemini (impacto alto, esfuerzo medio)
Gemini permite cachear porciones del prompt que se repiten (system prompt + knowledge-base) por hasta 1 hora. Ahorro estimado: **60-75% del costo de input tokens** en conversaciones largas.
- Cachear: system prompt + knowledge-base.md + JSON schema (todo lo estático)
- Enviar solo: historial reciente + mensaje actual
- Implementación: usar la API `cachedContent` del SDK de Gemini

### B. Resumen de historial antiguo (impacto alto, esfuerzo medio)
En vez de enviar los 12 turnos completos, comprimir los turnos N-5 a N-12 en un resumen breve de 200-300 tokens. Solo los últimos 4-5 turnos van textuales.
- Función `summarizeOldTurns(turns)` que mantiene contexto sin volumen
- Ahorro estimado: 30-40% en conversaciones largas

### C. Prompt slimming — recortar redundancias (impacto medio, esfuerzo bajo)
Auditar `gemini.service.js` y eliminar:
- Reglas duplicadas
- Ejemplos verbosos que pueden compactarse
- Instrucciones que ya están implícitas en el JSON schema
- Whitespace innecesario en el prompt
- Ahorro estimado: 15-20% en cada llamada (todas las llamadas)

### D. Knowledge-base condicional por estado (impacto medio, esfuerzo medio)
No inyectar la base de conocimiento completa siempre. Mostrar solo las secciones relevantes al estado actual:
- PERFILANDO: marcas + reglas de patente + reglas multi-vehículo
- CONFIRMANDO_COMPRA: métodos de pago + sucursales + horarios
- ESPERANDO_COMPROBANTE: métodos de pago + reglas de comprobante
- Ahorro estimado: 20-30% del peso del knowledge-base por llamada

### E. Asegurar que el routing Flash vs Pro funcione bien (impacto alto, esfuerzo bajo)
Auditar el código actual de routing dual-model en `gemini.service.js`. Casos donde se debe usar Flash (10x más barato):
- Saludo / consulta simple sin estado complejo
- Mensajes de acuse ("perfecto", "listo")
- Cuando no hay imágenes ni razonamiento de compatibilidad
Solo usar Pro cuando: hay imágenes, multi-vehículo, lógica de patente/VIN, primer turno de cliente nuevo.
- Ahorro estimado: si actualmente Pro hace 80% de las llamadas, bajar a 30-40% reduce costo ~50-60%

### F. Output schema más compacto (impacto bajo, esfuerzo bajo)
El JSON de salida tiene muchos campos opcionales con descripciones largas. Comprimir:
- Nombres de campos cortos (`mc` en vez de `mensaje_cliente`) — solo si compatible con el resto del código
- Descripciones más concisas
- Ahorro estimado: 5-10% del output

### G. Métricas de tokens en logs (impacto: instrumentación)
Antes de optimizar, instrumentar para medir el baseline:
- Loguear `input_tokens`, `output_tokens`, `cached_tokens`, `model_used` por cada llamada
- Dashboard admin: panel "Costo IA" con gasto por día, costo promedio por sesión, costo por venta cerrada
- Sin esto no se puede medir el impacto real de las optimizaciones

**Plan sugerido (en orden):**

| # | Acción | Impacto | Esfuerzo | Cuándo |
|---|---|---|---|---|
| 1 | Instrumentar métricas de tokens (G) | Medición | Bajo | Primero — necesario para medir las demás |
| 2 | Auditar routing Flash/Pro (E) | Alto | Bajo | Quick win |
| 3 | Prompt slimming (C) | Medio | Bajo | Quick win |
| 4 | Context caching de Gemini (A) | Alto | Medio | Tras tener métricas |
| 5 | Knowledge-base condicional (D) | Medio | Medio | Después |
| 6 | Resumen de historial antiguo (B) | Alto | Medio | Si conversaciones largas son frecuentes |
| 7 | Output schema compacto (F) | Bajo | Bajo | Opcional |

**KPI objetivo:**
Reducir costo promedio por sesión cerrada en **≥50%** sin degradar calidad de respuestas (validado con tests E2E equivalentes a los actuales).

**Decisión clave a tomar antes de empezar:**
¿El presupuesto actual de Gemini Pro permite 1-2 semanas de instrumentación + experimentación, o hay urgencia de bajar costos en días? Si urgencia → arrancar directo con E (routing) + C (slimming) que son quick wins de bajo esfuerzo.

## REQ-08 — Cliente Memory / Perfil persistido para personalización

**Prioridad:** Media (no es palanca de ahorro de costos, sino de UX/fidelización)
**Detectado:** 2026-05-05
**Tipo:** Feature de personalización para clientes recurrentes

**Importante:** Este REQ NO debe confundirse con REQ-07 (optimización de tokens). El ahorro real de tokens lo dan Context Caching + Routing Flash/Pro + Prompt Slimming. El customer profile aporta valor en **calidad de respuesta, personalización y fidelización**, no en costo. La justificación económica de este REQ está en aumentar conversión / retención, no en bajar el gasto de Gemini.

**Problema:**
Hoy cada conversación arranca "ciega": el agente no sabe si el cliente compró antes, qué vehículos tiene, cuál es su sucursal habitual ni su método de pago preferido. Pierde oportunidades de:
- Saludo personalizado ("Cliente recurrente desde 2024, 5 compras previas")
- Reconocimiento de vehículo conocido sin reconfirmar marca/modelo cada vez
- Sugerencias proactivas al vendedor ("Suele comprar filtros cada 3 meses, último pedido fue hace 4")
- Pre-llenar preferencias: sucursal habitual, método de pago, dirección de envío

**Diseño propuesto:**

### 1. Tabla `cliente_perfiles` (nueva o extender `clientes` existente)

```sql
CREATE TABLE IF NOT EXISTS cliente_perfiles (
    id SERIAL PRIMARY KEY,
    phone VARCHAR(30) UNIQUE NOT NULL,
    nombre VARCHAR(100),
    rut VARCHAR(20),
    -- Métricas agregadas
    compras_totales INTEGER DEFAULT 0,
    monto_total_gastado INTEGER DEFAULT 0,
    primera_compra_en TIMESTAMPTZ,
    ultima_compra_en TIMESTAMPTZ,
    -- Preferencias derivadas
    sucursal_habitual VARCHAR(30),         -- la más usada en sus compras
    metodo_pago_habitual VARCHAR(30),      -- transferencia | local | mixto
    metodo_entrega_habitual VARCHAR(30),   -- retiro | domicilio
    -- Vehículos conocidos (denormalizado para consulta rápida)
    vehiculos_conocidos JSONB DEFAULT '[]', -- [{marca_modelo, ano, patente?, vin?, ultima_vez_visto}]
    -- Repuestos comprados (para sugerencias)
    repuestos_historicos JSONB DEFAULT '[]', -- [{nombre, fecha, cantidad_total}]
    -- Notas del vendedor (manuales, opcionales)
    notas_internas TEXT,
    updated_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_cliente_perfiles_phone ON cliente_perfiles(phone);
```

### 2. Trigger / Job: poblado automático

Cada vez que se archiva un pedido (`archiveSession` → INSERT en `pedidos`), un hook actualiza el perfil:
- Incrementa `compras_totales`, `monto_total_gastado`
- Actualiza `ultima_compra_en`
- Ajusta `sucursal_habitual` (mode), `metodo_pago_habitual`, etc.
- Mergea vehículos al array (sin duplicar)
- Mergea repuestos a `repuestos_historicos`

Implementación: función `actualizarPerfilCliente(phone, pedido)` en backend, llamada al final de `archiveSession`.

### 3. Inyección al prompt como CONTEXTO PASIVO

El system prompt incluye un bloque NUEVO si `cliente_perfiles[phone]` existe:

```
👤 PERFIL DEL CLIENTE (solo lectura — NO copiar a entidades de esta sesión):
- Cliente recurrente desde {primera_compra_en}, {compras_totales} compras previas
- Sucursal habitual: {sucursal_habitual}
- Vehículos conocidos: {vehiculos_conocidos}
- Última compra: {ultima_compra_en}, monto {monto_total_gastado_total}

USO DE ESTE PERFIL:
- Saluda con familiaridad si aplica ("Hola Carlos, qué bueno verte de nuevo").
- Si menciona un vehículo de la lista de conocidos, NO le pidas confirmar marca/modelo.
- NO sugieras automáticamente repuestos del historial — solo cotiza lo que el cliente pide AHORA.
- NO copies vehiculos_conocidos a entidades.vehiculos[] — son DOS contextos distintos.
```

### 4. ⚠️ Reglas estrictas para evitar BUG-POST03 (lección aprendida hoy)

El profile debe ser **READ-ONLY desde el agente**. No debe:
- Auto-poblar `entidades.repuestos_solicitados[]` con repuestos históricos
- Asumir vehículo si el cliente no lo nombró en el mensaje actual
- Mergear `vehiculos_conocidos` con `entidades.vehiculos[]`

El BUG-POST03 detectado hoy (filtro de aceite que se coló sin que Carlos lo pidiera) viene precisamente de un merge agresivo. Aquí la regla es: **el perfil enriquece el contexto, no muta el estado**.

### 4-bis. Auto-derivación de sucursal para cliente recurrente (sub-feature de REQ-05)

Si el cliente recurrente ya tiene `sucursal_habitual` definida con suficiente confianza (≥3 compras previas Y ≥80% concentradas en una sucursal), el agente NO debe preguntar abierto "¿Melipilla o San Felipe?" sino **CONFIRMAR**:

> "Hola Carlos 🙌 ¿Igual que siempre, retiro en Melipilla?"

**Lógica de decisión:**
- Si `compras_totales >= 3` Y `concentracion(sucursal_habitual) >= 80%` → CONFIRMAR (1 turno menos)
- Si cliente confirma ("sí", "igual") → setear `sucursal_retiro = sucursal_habitual` y avanzar
- Si cliente niega ("no, esta vez quiero la otra", "San Felipe") → fallback a la pregunta abierta original
- Si cliente nuevo o histórico mixto (50/50) → pregunta abierta normal

**Persistencia:** la columna `sucursal_habitual` se recalcula al archivar cada nuevo pedido (función `actualizarPerfilCliente`). Si el cliente cambia su patrón habitual, la derivación se ajusta naturalmente.

**Beneficios concretos:**
- Ahorra 1 turno de conversación para clientes recurrentes (impacto directo en tokens — ~500-800 tokens por sesión)
- Mejora UX: cliente fiel siente que "lo conocen"
- Routing automático a la sucursal correcta sin fricción

**El mismo patrón aplica a:**
- `metodo_entrega_habitual` ("¿Igual que siempre, retiro en local?")
- `metodo_pago_habitual` ("¿Pagas con transferencia o efectivo como la última vez?")
- Estos NO son auto-aplicar el dato, sino **confirmar** para que el cliente pueda cambiar si quiere.

**⚠️ Cuidado de diseño:** la confirmación NO debe sentirse intrusiva. Si el cliente ya escribió "retiro en San Felipe" en su primer mensaje, NO confirmar contra el habitual — usar lo que dijo. La auto-derivación solo se aplica cuando el cliente NO mencionó la sucursal.

### 5. Vista admin: Perfil del cliente

Nueva sección en el dashboard al hacer click en un cliente:
- Tarjeta con `compras_totales`, `monto_total_gastado`, gráfica timeline de compras
- Lista de vehículos conocidos
- Top 5 repuestos comprados
- Botón "Editar notas internas" (texto libre, ej: "Mecánico de la flota X — preferir trato directo")

### 6. Compatibilidad con BUG-POST05 (saludo repetitivo)

Cuando el agente sepa que es cliente recurrente vía profile, el saludo personalizado del primer turno debe ser claro y único. Después del primer turno, no más saludos. Coordina con el fix de BUG-POST05.

**Cambios necesarios:**

1. **DB**: tabla `cliente_perfiles` (o extender `clientes`)
2. **Backend** `sessions.service.js`:
   - Función `actualizarPerfilCliente(phone, pedido)` ejecutada al archivar
   - Función `obtenerPerfilCliente(phone)` para inyectar al prompt
3. **Backend** `gemini.service.js`:
   - Cargar perfil al construir el prompt y agregarlo como bloque "PERFIL DEL CLIENTE"
   - Reglas estrictas de uso (no auto-merge)
4. **Backend** script de migración: poblar `cliente_perfiles` desde `pedidos` históricos
5. **Frontend** dashboard: vista de perfil cliente (vista admin opcional, posterior)

**KPIs objetivo:**
- Reducción de turnos por sesión para clientes recurrentes (ej: -2 turnos promedio)
- Aumento de NPS / reseñas positivas (más percepción de "me conocen")
- Reducción de errores de identificación de vehículo

**Lo que NO mide este REQ:**
- Costo de tokens — eso lo cubre REQ-07
- Aunque cliente_perfiles podría sumarse a una versión cacheada del prompt (Context Caching de REQ-07.A), eso es bonus, no el driver principal.

**Decisión de prioridad:**
Este REQ tiene MENOS urgencia que REQ-07 (que sí impacta caja directamente). Sugerir: implementar REQ-07 primero (instrumentación + quick wins), validar ahorro real, y luego abordar REQ-08 cuando se tenga tiempo para una iteración de UX/personalización.
